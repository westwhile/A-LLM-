# 仓库整理诊断与执行方案

> 盘点日期：2026-07-19
> 适用仓库：当前 `A-LLM-` 主仓库及其上级工作区
> 执行状态：2026-07-19 已完成本地整理；未推送远端。
> 原则：先保全当前成果，再合并入口；所有破坏性步骤均在快照、哈希或标签保护后执行。

## 一、结论

当前问题不是简单的“存在多个项目副本”，而是四类状态叠加：

1. 工作区根目录不是有效 Git 仓库；真正的 Git 仓库位于 `A-LLM-`。
2. 同级三个 `A-LLM-*` 历史目录是同一仓库的附加 worktree，不是可直接删除的普通副本。
3. 主线 worktree 有一批尚未提交的有效开发成果，同时混有生成报告、图表和文档交付物。
4. Git 忽略规则只解决“是否纳入版本控制”，没有解决百度同步盘仍在同步 `tmp`、虚拟环境和缓存的问题。

建议最终收口为：一个日常开发 worktree、一条清晰的 `main` 主线、历史提交以标签或归档分支保留、生成物与源代码分层存放。

## 三、建议的目标结构

```text
A股多因子选股策略研究/
├─ README.md                         # 仅说明如何进入主仓库
└─ A-LLM-/                          # 唯一日常开发 worktree
   ├─ src/                          # 源代码
   ├─ tests/                        # 测试
   ├─ config/                       # 配置与示例配置
   ├─ docs/
   │  ├─ plans/                     # 实施计划、模块规划
   │  └─ archive/                   # 只读历史说明，不放完整代码副本
   ├─ reports/
   │  ├─ advisor/                   # 导师报告源文件与正式交付物
   │  └─ releases/<run_id>/         # 选择性保留、可复现的正式图表快照
   ├─ data/sample/                  # 可公开、可复现的小样本
   ├─ data/raw/                     # 本地真实数据，Git 忽略
   ├─ outputs/                      # 运行产物，Git 忽略
   └─ tmp/                          # 仅临时使用，建议实际迁出同步盘
```

历史能力通过 Git 提交、标签和归档分支追溯，不再依赖三个长期挂载的工作目录。

## 四、执行顺序

### 阶段 0：冻结与保全

1. 暂停在三个历史 worktree 中继续开发。
2. 保留当前所有未提交修改、stash 和分支引用。
3. 对主线当前改动运行测试和最小研究流程，先判断哪些成果可组成稳定检查点。
4. 在任何清理前生成仓库 bundle，并单独备份未跟踪的重要文档和代码。

成功判据：当前成果存在至少两份可恢复副本，并记录当前提交、分支和文件清单。

### 阶段 1：把主线改动拆成可审查单元

建议按以下主题分组，不把所有 67 个状态项塞进一个提交：

1. 研究治理、配置契约、溯源、统计推断和执行合规；
2. 时间序列研究模块及其测试；
3. iFinD 能力探测、字段映射和权限说明；
4. 导师报告脚本、报告源文件与候选交付物；
5. 仓库卫生：依赖锁、换行策略、生成物边界和文档入口。

生成图表的变化要单独审查：若只是例行测试覆盖，不应混入功能提交；若是正式基线结果，则应绑定 `run_id`、配置哈希和证据清单后作为发布快照保存。

成功判据：主线测试通过，工作区干净，每个提交只表达一个可解释主题。是否执行 Git commit 和 push 需用户明确授权。

### 阶段 2：收口历史 worktree

1. 为三个独立历史提交建立明确的归档标签或归档分支；
2. 处理三个历史 worktree 中的 `README.md` / `ARCHIVED_READ_ONLY.md` 未提交状态；
3. 仅在确认可恢复后，用 `git worktree remove` 正常移除历史 worktree；
4. 将本地 `main` 与 `origin/main` 重新对齐；
5. 让 `A-LLM-` 成为唯一日常 worktree，并最终切换到 `main`。

成功判据：`git worktree list` 只保留主线，历史提交仍可通过标签或归档分支访问，`main` 不再分叉。

### 阶段 3：整理文档与运行产物

1. 复制并核对根目录三份重要 Markdown，再迁入主仓库；
2. 导师报告只保留“源 Markdown + 当前正式 DOCX/PDF + 对应运行证据”，旧候选版移入归档；
3. 日常运行只写 `outputs/runs/<run_id>/`；测试只写系统临时目录或任务临时目录；
4. `reports/figures/` 改为仅保存正式发布快照，避免被 smoke test 覆盖。

成功判据：重要文档均处于 Git 可追踪路径；重新运行测试不会修改正式报告资产。

### 阶段 4：建立防复发规则

1. 工作区根 README 只指向一个开发入口；
2. 在工作区或个人约定中写明“历史 worktree 禁止运行和修改”；
3. 在工作树干净后新增 `.gitattributes`，将换行规范化作为独立变更处理；
4. 明确 `pyproject.toml + uv.lock` 或其他唯一依赖基准；
5. CI 检查测试不得写入 `reports/`，并检查未提交生成物；
6. 尽量把 Git 元数据、虚拟环境和包缓存迁出百度同步盘，降低同步冲突和锁文件风险。

## 五、当前禁止直接执行的操作

在阶段 0 和阶段 1 完成前，不应执行：

- `git reset --hard`
- `git clean -fdx`
- 强制删除三个历史目录
- 强制移动或重置本地 `main`
- 清空 stash
- 批量覆盖根目录报告和主线报告
- 对整个仓库一次性执行换行重写

这些操作都有可能丢失尚未保存的代码、历史独立提交、研究报告或恢复线索。
