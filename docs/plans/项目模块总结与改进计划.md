# A股多因子选股策略研究：项目模块总结与改进计划

更新日期：2026-07-10
项目根目录：当前 `A-LLM-` 仓库
主线交付目录：`A-LLM-`

## 1. 总体判断

当前项目已经不是单纯的计划稿，而是一个可运行的 A 股多因子研究工程框架。主线目录 `A-LLM-` 已经覆盖：

- 合成样例数据生成、读取和标准化。
- 12 张标准表 schema：交易日历、股票基础、行情、估值/资金流、行业、指数成分、财务、停牌、ST、涨跌停、基准、事件。
- 数据质量报告与真实模式阻断检查。
- 量价、风险、估值、质量、成长、资金流、事件类因子。
- MAD 去极值、截面标准化、行业/市值中性化、覆盖率审计。
- IC、Rank IC、分组收益、年度 IC、滚动 IC、市场状态 IC、因子衰减和因子筛选报告。
- TopN 组合构建、下一交易日开盘事件式回测、订单/成交/持仓/现金审计。
- 佣金、印花税、滑点、冲击成本基础模型。
- 停牌、涨停禁买、跌停禁卖、手数、最大换手、最小成交额、成交额参与率限制。
- 基准收益严格日期对齐后的绩效、回撤、换手、行业暴露、成本归因和贡献分析。
- CLI、Notebook、Markdown/PDF 报告、图表、CSV 结果和单元测试。

但项目仍应定位为“研究工程框架和样例流程”，还不能表述为真实 A 股策略已经验证完成。主要原因是：真实数据链路仍不完整，AkShare 只覆盖了部分稳定端点，历史 PIT 表需要本地标准表补齐；因子权重和参数还没有完整 walk-forward；报告中部分样例图仍基于合成数据；LLM 事件标注客户端尚未合并进主线；配置、run 元数据和批量稳健性检验还不够完整。

后续开发原则：不要继续在多个同名包目录里并行改代码。所有新开发优先进入 `A-LLM-`，其他 `A-LLM-*` 目录只作为历史来源或待合并能力参考。

## 3. 主线模块总结

### 3.1 配置模块 `A-LLM-/config/`

| 文件 | 作用 | 已实现 | 问题与改进 |
|---|---|---|---|
| `project_config.yaml` | 项目名称、市场、基准、样本区间、信号/执行时间、股票池规则 | 已定义中证500、月末调仓、T 日收盘信号、下一交易日执行 | `run-pipeline` 目前主要读取因子配置，项目配置没有完整驱动数据区间、股票池和输出元数据 |
| `factor_config.yaml` | 因子启用列表和预处理参数 | 已覆盖 30 个左右候选因子和处理参数 | 因子方向仍是静态配置，缺少滚动训练期方向锁定和样本外约束 |
| `backtest_config.yaml` | 组合、成本、执行价格假设 | 已定义 Top50、5% 权重上限、成本和执行假设 | 主线 CLI 默认 `top_n=10`、`max_weight=0.2`，与配置文件不一致；成本模型运行时仍使用默认 `CostConfig()` |

改进要点：

- 统一配置入口，避免 CLI 默认值、配置文件和报告样例互相矛盾。
- 每次 `run-pipeline` 保存 `project_config.yaml`、`factor_config.yaml`、`backtest_config.yaml` 三个快照。
- 输出中记录代码版本、数据版本、run_id、股票池、样本期、因子列表、成本假设和执行假设。

### 3.2 数据模块 `A-LLM-/src/ashare_factor_research/data/`

| 文件 | 作用 | 已实现 | 问题与改进 |
|---|---|---|---|
| `schema.py` | 标准表 schema、主键、日期字段、字段校验 | 12 张标准表已定义 | 字段类型、单位、枚举值和跨表一致性校验仍可增强 |
| `data_loader.py` | 本地 CSV/Parquet loader 与 AkShare provider | 本地表读取可用；AkShare 支持交易日历、股票基础、个股日行情、基准指数 | `daily_basic`、历史指数成分、行业、ST、停牌、涨跌停仍要求本地标准表补齐 |
| `data_quality.py` | 数据质量审计与阻断 | 已检查缺表、主键重复、价格非正、OHLC 越界、财务 PIT 错误、基准价格 | 缺少成交额异常、复权因子跳变、覆盖率阈值、行业/股票池连续性、跨表日期一致性阻断 |
| `data_cleaner.py` | 复权价格、未来收益、股票池过滤 | 已处理 adj price、forward return、ST/停牌/历史成分等 | 真实退市、新股上市天数、行业历史变更、涨跌停状态和长期停牌仍需更细 |
| `trading_calendar.py` | 交易日、下一交易日、月末调仓日 | 已可用 | 真实节假日和 provider 日历差异需要缓存与验证 |
| `sample_data.py` | 合成样例数据 | 已覆盖主线流程所需字段 | 只能验证工程链路，不可用于策略结论 |

数据模块是项目中最重要的风险源。真实研究必须明确区分 `trade_date`、`ann_date`、`usable_date`、`publish_date`、`signal_date`、`execution_date` 和 `target_return_end_date`。任何把未来公告、未来成分、未来行业分类或未来交易状态提前用于信号日的处理，都会污染因子检验和回测。

### 3.3 因子模块 `A-LLM-/src/ashare_factor_research/factors/`

| 文件 | 作用 | 已实现 | 问题与改进 |
|---|---|---|---|
| `registry.py` | 因子注册表、方向、类别、窗口、输入字段、PIT 标记、中性化方式 | 已覆盖量价、风险、基本面、质量、成长、资金流、事件类因子 | 方向仍是先验/静态设定，需要训练期锁定；因子说明应进入最终报告 |
| `price_volume_factors.py` | 动量、反转、换手、成交额、Amihud 等 | 已实现 | Beta/idio vol 使用市场代理，真实基准或行业模型需进一步确认 |
| `fundamental_factors.py` | BP、EP、ROE、ROA、毛利率、成长、现金流等 | 已实现 PIT 对齐思路 | 真实财报的修订值、TTM、缺失、公告延迟和字段单位需要真实数据验证 |
| `money_flow_factors.py` | 资金流强度因子 | 已实现 | 真实资金流字段来源和可得性需校验，不能默认所有股票全覆盖 |
| `llm_event_factors.py` | 事件情绪因子聚合 | 已实现结构化事件到因子的聚合 | 主线没有真实 LLM 标签生成、缓存、prompt 版本和人工抽查模块 |
| `factor_processor.py` | 去极值、标准化、中性化、相关性、处理审计 | 已实现 | 中性化样本不足时需要更严格降级规则；覆盖率低的因子应阻断或退出组合 |
| `coverage.py` | 覆盖率、行业覆盖、市值分桶、缺失 streak | 已实现 | 结果目前偏诊断输出，尚未完全成为硬门槛 |

改进优先级不是继续堆新因子，而是把每个已启用因子的时间语义、字段来源、方向确定、覆盖率门槛、分年度稳定性和样本外表现做扎实。

### 3.4 因子检验模块 `A-LLM-/src/ashare_factor_research/factor_testing/`

| 文件 | 作用 | 已实现 | 问题与改进 |
|---|---|---|---|
| `ic_test.py` | IC、Rank IC、ICIR、t 值、胜率 | 已实现 | 真实研究中目标收益必须与信号可用时间严格错开 |
| `group_test.py` | 分组收益、分组累计收益、多空差 | 已实现 | A 股做空约束下，多空差不能直接等同可交易收益 |

| `factor_selection.py` | 因子筛选报告 | 已实现 | 目前仍是研究输入，不能作为样本外有效性证明 |

必须补的研究流程：

- 训练期确定因子方向和入选规则。
- 验证期选择窗口、权重、调仓频率和成本假设。
- 测试期只做一次最终评估。
- 报告中区分样本内表现和样本外表现。

### 3.8 LLM 辅助模块

主线目前只有 `factors/llm_event_factors.py`，能把已经结构化的事件标签聚合为因子。完整 LLM 标注工程还在 `A-LLM-portfolio-backtest-only/src/ashare_factor_research/llm/`，包括：

- `client.py`：离线 rule-based labeler、批量标注、重试、缓存接入。
- `prompts.py`：prompt 模板和版本。
- `schema.py`：标签字段校验和规范化。
- `cache.py`：JSONL 标签缓存。
- `audit.py`：抽样人工复核报告。

主线缺口：

- 没有真实文本抓取或导入标准。
- 没有 prompt 版本、模型版本、缓存 key、原文、输出标签的完整审计链。
- 没有人工抽样复核结果进入质量门槛。
- 事件发布时间与信号截止时间的约束还需要在真实数据里强制检查。

LLM 模块应定位为事件结构化和辅助解释，不应默认作为交易指令来源。

## 4. 当前主要未实现和需改进功能

### P0：影响真实研究正确性的缺口

1. 真实数据闭环未完成。
   - AkShare provider 只保守支持交易日历、股票基础、个股日行情和基准指数。
   - `daily_basic`、历史指数成分、行业、ST、停牌、涨跌停、完整财务表仍需本地标准表。
   - 需要形成“拉取/导入 -> 标准化 -> 质量阻断 -> 缓存版本 -> pipeline”的闭环。

2. Point-in-time 和历史股票池还需真实数据验证。
   - 财务数据必须使用 `usable_date`。
   - 指数成分、ST、停牌、行业分类、涨跌停和退市状态都必须是当时可见。
   - 当前样例测试不能证明真实数据不存在未来函数。

3. 配置驱动不一致。
   - `backtest_config.yaml` 的 Top50/5% 与 CLI 默认 Top10/20% 不一致。
   - 成本配置文件未完全进入 `run_event_backtest`。
   - run 目录只快照了项目配置，没有完整快照 factor/backtest 配置。

4. 样本外研究流程缺失。
   - 因子方向、权重、筛选规则、调仓频率和成本假设不能用全样本反复调。
   - 需要 walk-forward 或明确训练/验证/测试切分。

5. 报告状态需要同步。
   - `reports/factor_research_report.md` 中仍有“样例基准为现金占位”的旧表述。
   - 主线 `run-pipeline` 已支持真实基准严格对齐，应在报告中区分 sample 图和 real run。

### P1：影响回测可信度和展示质量的缺口

1. 成交可达性需要真实市场规则。
   - 当前 `constraints.py` 仍是简化版。
   - 需要交易所涨跌幅规则、ST 涨跌幅、科创/创业板规则、退市整理、长期停牌、复牌首日等处理。

2. 容量和流动性分析不系统。
   - 虽然引擎支持 `max_participation_rate` 和 `min_trade_amount`，但 pipeline 没有批量情景报告。
   - 需要按 1%、5%、10% 成交额参与率做压力测试。

3. 稳健性检验不足。
   - 缺少 Top30/Top50/Top100、周/月/双月、成本 0/标准/高成本、延迟成交、不同股票池的自动对比。

4. 归因还不够深入。
   - 已有行业、市值、个股和成本归因。
   - 还需要主动行业暴露、风格暴露、回撤区间归因、因子收益归因。

5. LLM 标注工程未并入主线。
   - 当前只消费标签，不生产可审计标签。
   - 需要 dry-run、缓存、版本、抽样复核和标签一致性。

### P2：影响可维护性和复现性的缺口

1. 阶段性目录尚未归档。
   - 同名包存在于多个目录，容易误改。
   - 建议合并 `llm/` 后将阶段性目录移入 `archive/` 或写只读说明。

2. CLI 还不完整。
   - 建议增加 `version`、`build-report`、`run-robustness`、`describe-run`。

3. 测试偏样例。
   - 应增加 provider mock、数据质量边界、配置解析、报告产物完整性、真实模式缺表阻断、交易限制边界。

4. 文档分层需要更清楚。
   - README 讲快速运行。
   - `implementation_status.md` 讲当前状态。
   - 本文件讲全局模块和路线图。
   - 研究报告只讲研究逻辑和结果，避免混入过多工程计划。

## 5. 详细改进计划

### 阶段 1：配置统一和状态同步

目标：让项目运行参数、报告表述和输出元数据一致。

任务：

1. 修改 `run-pipeline`，支持读取 `project_config.yaml`、`factor_config.yaml`、`backtest_config.yaml`。
2. 统一 CLI 默认值与配置文件：
   - 默认 Top50。
   - 单票最大权重 5%。
   - 成本参数从 `backtest_config.yaml` 注入 `CostConfig`。
3. run 目录保存完整配置快照：
   - `project_config_snapshot.yaml`
   - `factor_config_snapshot.yaml`
   - `backtest_config_snapshot.yaml`
4. 更新 `reports/factor_research_report.md`，区分：
   - sample 模式：合成数据、工程验证。
   - real 模式：真实数据、基准严格对齐。
5. 增加 `version` CLI，输出包版本、Python 版本、pandas 版本、配置路径和当前工作目录。

验收标准：

- `python -m ashare_factor_research.main run-pipeline --mode sample --run-id sample-smoke` 的组合参数与配置文件一致。
- `outputs/runs/<run_id>/` 中保存完整配置快照。
- 报告不再把旧的现金占位基准表述混同为当前全部 pipeline 状态。

### 阶段 2：真实数据标准化闭环

目标：让项目能从真实数据或本地标准表稳定进入研究流程。

任务：

1. 明确两条数据入口：
   - AkShare 可稳定拉取的表：`trade_calendar`、`stock_basic`、`daily_bar`、`benchmark_index`。
   - 必须本地补齐的 PIT 表：`daily_basic`、`industry`、`index_member`、`financial_indicator`、`suspension`、`st_status`、`limit_price`、`news_event`。
2. 增加 `data/import_standard.py` 或 CLI `import-data`：
   - 检查列名映射。
   - 转换日期。
   - 写入 Parquet。
   - 生成 `data_manifest.json`。
3. 扩展数据质量检查：
   - 交易日覆盖率。
   - 股票覆盖率。
   - 成交额为 0 或异常尖峰。
   - 复权因子跳变。
   - 行业缺失。
   - 指数成分区间重叠。
   - 同一股票 ST/停牌区间重叠。
4. 对真实模式设置 blocking 规则：
   - 缺少核心表失败。
   - 主键重复失败。
   - PIT 错误失败。
   - 价格和 OHLC 错误失败。
   - 基准与策略日期无法严格对齐失败。

验收标准：

- 能对一个小股票池真实数据目录运行 `quality-check --mode real --fail-on-blocking`。
- 数据质量报告能明确列出缺表、缺字段、重复主键和 PIT 风险。
- 真实数据进入 pipeline 前必须生成 manifest。

### 阶段 3：PIT 和防泄漏专项

目标：优先把未来函数和生存者偏差风险压低。

任务：

1. 在因子面板保留并输出：
   - `signal_date`
   - `execution_date`
   - `target_return_end_date`
   - 财务数据的 `ann_date`/`usable_date` 来源标记。
2. 对财务因子增加单元测试：
   - 公告日前不可见。
   - 公告日后下一交易日可见。
   - 多期财报只取 signal date 前最新可用值。
3. 对股票池增加测试：
   - 历史指数成分按 `in_date/out_date` 生效。
   - ST 区间内剔除。
   - 停牌日不可交易。
   - 退市后不可进入股票池。
4. 对事件因子增加测试：
   - `publish_date` 晚于 signal date 的事件不进入。
   - 同一事件重复标签去重。

验收标准：

- PIT 测试覆盖财务、指数成分、ST、停牌和事件。
- 任意真实模式 run 的因子面板可以追溯信号和目标收益时间窗口。

### 阶段 4：样本外因子研究

目标：从“全样本演示”升级为“可解释的样本外研究”。

任务：

1. 新增 walk-forward 配置：
   - 训练窗口：例如过去 24 个月。
   - 验证窗口：例如过去 6 个月。
   - 测试窗口：当前下一个调仓月。
2. 在训练窗口确定：
   - 因子方向。
   - 入选因子。
   - 因子权重。
   - 覆盖率门槛。
3. 在测试期固定规则运行，不用未来 IC 调整。
4. 输出：
   - 样本内 IC。
   - 样本外 IC。
   - 因子方向变化记录。
   - 入选因子历史。
   - 因子权重历史。
5. 对异常结果自动标记：
   - IC 绝对值过高。
   - 覆盖率过低。
   - 单日期贡献过大。
   - 单行业贡献过大。
   - 与未来收益字段异常相关。

验收标准：

- 报告能明确说明因子方向和权重来自过去窗口。
- 样本外结果与样本内结果分开展示。
- 不能再用全样本 ICIR 直接决定回测期权重。

### 阶段 5：回测执行和容量压力测试

目标：让回测更接近 A 股真实交易约束。

任务：

1. 扩展交易约束：
   - 主板、创业板、科创板不同涨跌幅。
   - ST 涨跌幅。
   - 新股和退市整理特殊处理。
   - 长期停牌和复牌首日处理。
2. 增加批量执行情景：
   - 无成本。
   - 标准成本。
   - 高成本。
   - 延迟 1 日成交。
   - 延迟 2 日成交。
   - 成交额参与率 1%、5%、10%。
3. 输出未成交分析：
   - 未成交订单数。
   - 未成交金额。
   - 未成交原因分布。
   - 被涨停/跌停/停牌阻断的收益影响。
4. 增加组合约束：
   - 行业偏离上限。
   - 最小持仓数量。
   - 单行业上限。
   - 最大现金比例或现金处理规则。

验收标准：

- 每个调仓日能追溯目标权重、订单、成交、未成交原因和持仓延续。
- 报告展示成本、延迟、容量三类压力测试下的绩效变化。

### 阶段 6：风险归因和报告升级

目标：让报告能解释收益来源、风险来源和成本损耗。

任务：

1. 在有真实基准时输出：
   - 策略收益。
   - 基准收益。
   - 超额收益。
   - Alpha、Beta、跟踪误差、信息比率。
   - 超额最大回撤。
2. 增加归因：
   - 行业主动暴露。
   - 行业收益贡献。
   - 市值分桶收益贡献。
   - 个股 Top/Bottom 贡献。
   - 回撤区间归因。
   - 成本拖累拆解。
3. 增加稳健性图表：
   - 年度收益表。
   - 月度收益热力图。
   - 换手率时间序列。
   - 成本情景对比。
   - 容量情景对比。
   - IC 衰减图。
4. 报告强制披露：
   - 数据来源和版本。
   - 样本区间。
   - 股票池。
   - 调仓规则。
   - 成本假设。
   - 交易限制。
   - 未解决偏差。

验收标准：

- 报告不只展示净值曲线，还能说明收益来自哪里、回撤来自哪里、成本损耗多少。
- 没有真实基准时，不输出或不解读 Alpha/IR 等相对指标。

### 阶段 7：LLM 事件模块合并

目标：把 LLM 做成可审计的事件标签工程，而不是黑箱预测器。

任务：

1. 从 `A-LLM-portfolio-backtest-only` 合并：
   - `llm/client.py`
   - `llm/prompts.py`
   - `llm/schema.py`
   - `llm/cache.py`
   - `llm/audit.py`
2. 默认 dry-run，不默认调用外部 API。
3. 每条标签保存：
   - 原始文本。
   - 文本来源。
   - 发布时间。
   - prompt 版本。
   - 模型版本。
   - 输出 JSON。
   - cache key。
   - 创建时间。
4. 每次 run 抽样生成审计表：
   - `review_status`
   - `error_type`
   - `review_comment`
5. 事件因子只使用 `publish_date <= signal_date` 的标签。
6. 标签质量低于阈值时，事件因子不进入组合，只进入解释章节。

验收标准：

## 6. 推荐实施顺序

| 顺序 | 工作 | 原因 |
|---|---|---|
| 1 | 统一配置驱动和报告状态 | 先消除当前最容易误导的参数/文档不一致 |
| 2 | 补齐真实数据标准化和质量阻断 | 没有可信数据就不能谈因子有效 |
| 3 | 做 PIT 防泄漏专项测试 | 量化研究最关键的正确性风险 |
| 4 | 建立 walk-forward 因子研究流程 | 防止全样本筛选和重复调参 |
| 5 | 扩展回测交易限制、容量和成本情景 | 提高回测可执行性可信度 |
| 6 | 升级风险归因和报告 | 让结果可解释，而不是只看收益曲线 |
| 7 | 合并 LLM 标注与审计模块 | 完成项目特色，但不能早于数据治理 |
| 8 | 归档阶段性目录和完善测试/CI | 降低维护成本 |

## 7. 开发验收清单

每完成一个版本至少检查：

- 数据是否有 schema、主键唯一和质量报告。
- 真实模式缺核心表是否阻断。
- 财务、事件、指数成分、ST、停牌、涨跌停是否按当时可见时间进入。
- 信号日、执行日、目标收益结束日是否写入输出。
- 因子覆盖率、行业覆盖、市值分桶覆盖和缺失 streak 是否输出。
- 因子方向和权重是否避免全样本事后选择。
- 组合是否满足持仓数量、单票权重、行业/流动性/换手约束。
- 回测是否输出订单、成交、持仓、现金和未成交原因。
- 成本、滑点、成交限制、容量和延迟成交是否有压力测试。
- 基准收益是否严格日期对齐。
- 报告是否明确样本区间、股票池、数据版本、成本假设和未解决偏差。
- 测试是否覆盖核心计算逻辑和关键边界。

## 8. 结论

当前主线 `A-LLM-` 已经具备较完整的量化研究工程骨架，代码模块、报告、图表、run 输出和测试都已经形成闭环。下一步不要优先追求更多因子或更漂亮的样例收益，而应优先解决真实数据标准化、PIT 防泄漏、配置一致性、样本外验证、交易可达性、容量压力测试和报告状态同步。

在这些环节完成前，所有回测收益都只能表述为研究框架验证结果，不能表述为真实可获得收益，也不构成任何投资建议。
