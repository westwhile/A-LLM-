# A股多因子项目专业时间序列研究执行计划

## 一、目标、范围与当前起点

### 1. 总目标

建立一套严格样本外、可复现、可审计的专业时间序列研究体系，回答三个核心问题：

1. 因子有效性是否存在可预测的时间漂移？
2. 市场状态是否能够改善因子配置和回撤控制？
3. 条件波动预测是否能够改善组合风险收益比？
4. 数据、模型、回测和审计证据是否能够通过一个只读研究看板被持续追踪和复核？

最终策略结构为：

```text
截面因子分数
× 动态因子权重
× 市场状态调整
× 波动率目标缩放
→ 结构化研究证据
→ 只读前端看板
```

### 2. 第一阶段边界

第一阶段不将以下模型投入主生产研究：

- PatchTST、TCN 等深度网络；
- 全股票 VAR/VECM；
- 全股票 DCC-GARCH；
- 高频日内模型；
- 大规模自动超参数搜索；
- 直接预测股票价格水平；
- 逐股票独立训练深度模型。

### 3. 当前工程起点

当前项目已经具备：

- 时间序列配置契约；
- 月度因子 IC 和因子收益构造；
- HMM、Kalman、GJR-GARCH 和 DCC 第一版；
- DM、SPA、DSR 和 PBO 审计；
- `fit/update/forecast` 统一接口；
- 真实模式历史准入和禁止降级机制；
- 共同投资区间绩效比较；
- 67 项通过的自动化测试。

当前尚未完成：

- 2015 年至今的真实 PIT 数据接入；
- 至少 36 个月可用的动态方案 OOS 结果；
- 模型稳定性和超参数研究；
- 三方案真实消融比较；
- 最终模型晋级结论。
- Kimi CLI/Kimi Code CLI 已完成本机安装、登录和版本冻结；私有仓库读取仍待明确的数据外发授权；
- 同花顺 iFinD 数据接口的账号权限、历史深度、字段和授权范围核验；
- 前端看板目录、数据契约和验收基线。

截至 2026-07-19 的复核已确认 Kimi Code CLI `0.27.0` 可执行、配置有效，且无项目数据的非交互结构化请求成功；因第三方服务数据外发风险，私有仓库读取未获授权。iFinD SDK、接口凭据和账号授权仍未发现，因此同花顺真实数据接入仍属于**待验证实施路径**，不得表述为已经可用。

## 三、阶段 0：冻结研究协议与基线

### 目标

在接入真实数据前固定研究对象、时间区间、模型清单和晋级标准，避免看到结果后修改研究规则。

### 执行任务

- [ ] 在研究协议中固定数据开始日、评价开始日和最终留出期。
- [ ] 固定预测频率为月度、预测步长为 1 个月。
- [ ] 固定因子择时、状态识别和波动模型候选清单。
- [ ] 固定 DM、SPA、DSR、PBO 的使用规则。
- [ ] 建立试验登记表，每个模型或参数组合分配唯一 `experiment_id`。
- [ ] 冻结当前合成数据基线，禁止覆盖其运行目录。

建议协议配置：

```yaml
research:
  data_start: "2015-01-01"
  evaluation_start: "2018-01-01"
  final_holdout_start: "2024-01-01"
  frequency: monthly
  forecast_horizon_months: 1
  minimum_oos_months: 36

model_registry:
  factor_timing:
    - static_equal
    - rolling_ic_12m
    - rolling_ic_24m
    - ewma_ic
    - kalman_local_level
  regime:
    - no_regime
    - rule_based
    - hmm_2_state
    - hmm_3_state
  volatility:
    - no_scaling
    - historical_vol
    - ewma
    - gjr_garch

selection:
  primary_metric: net_information_ratio
  prediction_tests:
    - dm
    - spa
  overfit_tests:
    - dsr
    - pbo
```

### 预注册研究假设

- **H1：**Kalman 动态权重扣费后优于静态等权和滚动 IC。
- **H2：**实时 HMM 状态调整可以降低回撤，而不显著损害长期收益。
- **H3：**GJR-GARCH 波动目标可以降低实际波动偏离和尾部损失。

### 验收标准

- [ ] 协议包含全部模型、窗口和评价门槛。
- [ ] 后续新增模型必须登记试验编号。
- [ ] 当前样例结果明确标记为 `synthetic_fallback`。
- [ ] 样例结果不得用于真实市场有效性结论。

- [ ] 运行一个只读提示词，验证 Kimi 能读取仓库、输出结构化结果且不修改文件。
- [x] 验证非交互 `-p/--prompt` 模式及其退出码；未使用 `--yolo`。当前版本不允许 `--plan` 与 `--prompt` 组合。
- [x] 本阶段未采用 MCP；如后续采用，须先登记服务器来源、版本、权限、可调用工具和数据去向。
- [x] 把 CLI 版本和执行模式写入 `reports/kimi_cli_capability_report.md`。


若当前版本不支持某个命令，必须以 `kimi --help` 的实际输出为准更新命令，不得照抄过时示例。[MoonshotAI 官方仓库](https://github.com/MoonshotAI/kimi-cli)已说明旧 Kimi CLI 正在迁移到新的 Kimi Code CLI，因此项目必须记录实际可执行文件版本，不能只记录产品名称。

### 2. 同花顺 iFinD 权限核验

根据[同花顺 iFinD 数据接口官方说明](https://quantapi.51ifind.com/gwstatic/static/ds_web/quantapi-web/help-center/faq.html)，正式接口可采用两条路径：

1. **Python SDK**：登录后调用 `THS_*` 函数；
2. **HTTP API**：使用授权令牌调用正式 `quantapi.51ifind.com` 接口。

实施前必须由用户确认：

- [ ] 是否已有 iFinD 数据接口账号，而不只是普通网站或客户端账号；
- [ ] 账号属于免费、试用还是正式权限；
- [ ] A 股行情、复权、财务、历史成分、ST、停牌、涨跌停和公告数据各自的历史深度；
- [ ] 单次、每日和总数据量限制；
- [ ] 是否允许本地缓存、研究使用、图表展示和导出衍生结果；
- [ ] 是否存在同账号并发登录限制；
- [ ] 令牌或账号密码的保管方式。

凭据只允许来自环境变量、操作系统凭据库或本机未跟踪配置。日志、提示词、截图、错误堆栈和 Kimi 会话导出中都必须脱敏。

如果用户没有已授权的数据接口账号，Kimi 的工作必须止于官方文档调研、字段缺口登记、适配器骨架和脱敏模拟响应测试，不得声称已经取得同花顺真实数据。

### 3. 最小能力探针

在下载全量历史数据前，由 Kimi CLI 生成并运行最小探针，范围限制为：

```text
2–3 只股票
2–3 个交易日
每类数据 1–3 个字段
不批量遍历指标
不绕过账号权限或频率限制
```

至少验证以下正式入口：

| 数据缺口 | iFinD 候选入口 | 最小验证内容 | PIT/合规重点 |
| --- | --- | --- | --- |
| 历史日行情 | `cmd_history_quotation` / SDK 历史行情函数 | OHLCV、成交额、缺失交易日 | 复权口径、停牌日处理 |
| 基础与财务字段 | `basic_data_service`、`date_sequence` / `THS_BD`、`THS_DS` | 指标 ID、报告期、取值日期 | 公告日、修订版、实际可用日是否可得 |
| 历史股票池/板块 | `data_pool` 或授权的板块成分函数 | 指定历史日成分 | 是否真为历史快照，防止当前成分回填 |
| 交易日 | `get_trade_dates` / SDK 日期函数 | 交易日与偏移 | 与项目交易日历对齐 |
| 公告信息 | `report_query` / `THS_ReportQuery` | 公告日、发布时间、唯一编号、链接 | 发布时间时区、盘后可用规则、修订追踪 |
| 宏观/状态变量 | `edb_service` / `THS_EDB` | 指标 ID、发布日期、历史值 | 初值与修订值、发布滞后 |
| 问财语义查询 | `smart_stock_picking` / `THS_WCQuery` | 仅做字段发现和人工核查 | 不作为正式 PIT 主数据源，不直接进入回测 |

指标 ID 和专题报表 ID 必须来自账号可见的 iFinD/SuperCommand 指标工具或官方文档，并写入字段映射；严禁让 Kimi 猜测指标 ID。

### 4. 数据源决策规则

对原计划中无法从开源渠道稳定取得的字段，按以下顺序处理：

```text
A. 官方或可合法开源取得，且可满足 PIT
   → 优先使用并保留来源快照

B. 开源不足，但用户的 iFinD 授权可覆盖
   → 由 Kimi CLI 生成适配器和探针
   → Codex 复核字段、时间、权限和证据
   → 通过后再批量导入

C. iFinD 有字段但缺公告日、修订历史或历史成分语义
   → 标记 PIT 不合格
   → 不得用当前值反填历史

D. 无权限、超配额、授权不允许缓存/展示，或来源仍不可核验
   → 标记 `blocked_by_data_license` 或 `unavailable`
   → 缩小研究问题，不得让 Kimi 用推测值补齐
```

不得通过网页抓取、自动化登录、逆向私有接口、共享账号或规避频控来替代正式授权。

## 四、阶段 1：真实 PIT 数据接入与质量审计

> 这是当前项目的关键路径。数据门槛未通过前，不进入正式模型比较。

- `trade_date`、`ts_code`；
- 开、高、低、收和前收盘价；
- 成交量、成交额；
- 复权因子；
- 停牌标记；
- 涨跌停价格；

- 上市日期和退市日期；
- ST 状态有效区间；

- 修订版本；
- 原始值；

## 五、阶段 2：构建月度时序研究样本

### 1. 建模对象

建立六类月度序列：

1. 月度因子 Rank IC；
2. 月度因子 Q5−Q1 收益；
3. 中证 500 月度对数收益；
4. 月度实现波动率；
5. 市场宽度；
6. 市场流动性。

不直接对股票价格水平进行 ARIMA 或 HMM 建模。

### 2. 月度因子 IC

每个有效月末依次执行：

1. 计算截面因子值；

## 七、阶段 4：Kalman 动态因子权重

> 这是第一优先研究模块。

### 1. 模型定义

对每个因子的月度 IC 建立局部水平模型：

$$
IC_{i,t}=\mu_{i,t}+\epsilon_{i,t}
$$

$$
\mu_{i,t}=\mu_{i,t-1}+\eta_{i,t}
$$

其中：

- $\mu_{i,t}$：潜在因子有效性；
- $\epsilon_{i,t}$：观测噪声；
- $\eta_{i,t}$：因子有效性漂移。

### 2. 权重构造

第一版权重分数：

$$
s_{i,t}=\frac{\hat{\mu}_{i,t}}{\hat{\sigma}_{i,t}^{2}+\lambda}
$$

依次执行：

1. 因子方向调整；
2. FDR 过滤；
3. 覆盖率过滤；
4. 单因子权重上限；
5. 权重归一化；
6. 换手惩罚。

换手平滑：

$$
w_t=(1-\rho)w_t^{raw}+\rho w_{t-1}
$$

默认 $\rho=0.2$。

### 3. 预注册参数网格

```yaml
process_variance:
  - 0.0001
  - 0.001
  - 0.01

observation_variance:
  - 0.005
  - 0.01
  - 0.02

turnover_penalty:
  - 0.0
  - 0.2
  - 0.5
```

27 组参数必须全部登记为试验次数。

### 4. 对照组

- 静态等权；
- 方向锁定后的等权；
- 12 个月滚动 IC；
- 24 个月滚动 IC；
- EWMA IC；
- Kalman IC。

### 5. 输出

```text
dynamic_factor_weights.csv
factor_weight_turnover.csv
factor_weight_stability.csv
factor_ic_forecasts.csv
factor_timing_comparison.csv
```

### 晋级标准

- [ ] 至少 36 个月非重叠 OOS。
- [ ] 扣费后优于静态等权。
- [ ] 增量收益不只依赖单一因子。
- [ ] 多数自然年度增量为正。
- [ ] 权重换手处于可交易范围。
- [ ] DM 或 SPA 至少一项支持预测改善。
- [ ] DSR 和 PBO 未显示严重数据挖掘风险。

## 八、阶段 5：HMM 市场状态识别

### 1. 第一版输入变量

```text
benchmark_log_return
realized_volatility
market_breadth
log_market_amount
```

所有变量只能在训练窗口内标准化。

### 2. 实验组

- 无状态；
- 规则式状态；
- 两状态 Gaussian HMM；
- 三状态 Gaussian HMM；
- 三状态 Student-t HMM，作为后续稳健性实验。

### 3. 实时估计流程

每个调仓点：

1. 截取截至当日的扩展训练窗口；
2. 在训练数据上拟合 HMM；
3. 根据收益、波动率和宽度给状态排序；
4. 只计算 filtered probability；
5. 输出下一持有期可使用的状态概率；
6. 禁止将 smoothed probability 用作策略输入。

### 4. 状态概率映射

采用软概率而不是硬切换：

$$
a_t=1-\lambda_bP(\text{bear}_t)+\lambda_uP(\text{bull}_t)
$$

第一版限制：

```text
0.90 <= a_t <= 1.00
```

### 5. 稳定性检查

- [ ] 不同训练窗口的状态含义稳定。
- [ ] 状态没有异常高频跳变。
- [ ] 状态持续期具有经济合理性。
- [ ] 状态概率没有长期退化到 0 或 1。
- [ ] 不同初始化结果基本一致。
- [ ] 两状态和三状态模型具有可解释的对应关系。

### 6. 输出

```text
regime_probabilities.csv
regime_transition_matrix.csv
regime_duration.csv
regime_factor_performance.csv
regime_stability_audit.csv
```

### 晋级标准

- [ ] filtered probability 不存在未来依赖。
- [ ] 状态含义在多数滚动窗口中稳定。
- [ ] 回撤或尾部损失得到改善。
- [ ] 改善不只依赖单次危机。
- [ ] 状态调整没有造成成本和换手失控。

